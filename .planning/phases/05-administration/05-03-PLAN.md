**ToggleSwitch.tsx:**
- Props: `{ checked: boolean; onChange: (value: boolean) => void; disabled?: boolean }`
- Renders an accessible toggle switch using `<button role="switch" aria-checked={checked}>`
- Styling: `h-6 w-11 rounded-full` container. When checked: `bg-accent`, unchecked: `bg-white/20`. Inner knob: `h-4 w-4 rounded-full bg-white` with `translate-x-6` (checked) or `translate-x-1` (unchecked). Transition on both color and transform.
- Disabled state: `opacity-50 cursor-not-allowed`

**CriteriaSection.tsx:**
- Uses `useQuery(adminQueries.ratingCriteria())`
- Columns for AdminTable:
  1. name (text input, editable)
  2. description (text input, editable)
  3. active (renders ToggleSwitch -- NOT part of edit mode, always interactive)
- The active column uses a custom `render` that shows ToggleSwitch. On toggle change, immediately call `toggleRatingCriterion(id)` mutation (no edit/save flow needed for toggle -- it's an instant action per CONTEXT.md).
- `onCreate` calls `createRatingCriterion({ name, description })` -- new criteria default to active (backend decides)
- `onSave` calls `updateRatingCriterion(id, { name, description })` -- toggle is separate
- `onDelete` calls `deleteRatingCriterion(id)` mutation
- After ANY mutation: invalidate both `adminQueries.ratingCriteria()` AND user-facing rating queries `queryClient.invalidateQueries({ queryKey: ['rating-criteria'] })` to keep user-facing rating forms in sync (Pitfall 3 from research)
- Search placeholder: "Search criteria..."
